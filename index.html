Serverless Computing For Every Developer

What is it?

The Distributed Compute Protocol is a lightweight, powerful framework for serverless applications. It is the ultimate tool for parallel computing, which enforces every CPU and GPU to follow the same simple programming model. DCP's platform lets the same code run on any device, network, or OS. With support for dynamic resource sharing across heterogeneous systems, it is easier than ever to create computer clusters at any scale. As a developer framework, DCP improves the quality of your apps through:

Simplified Programming

The DCP Compute API isthe easiest way to express parallel workloads. It is built by the same developers who use it everyday.

Fungible Infrastructure

Compute and network resources are 100% abstracted from hardware and optimized for your code, so you always get seamless performance.

Virtual Clusters

DCP uses ultra-lightweight web technology for almost no overhead, and better performance than apps written for Containers or VMs.

Compute Wherever, However.

In addition to your own hardware, compute grids from universities, data centers, and more are publicly available through DCP. Developers can execute work both locally and as a web service with the following deployment models:

Compute Groups

Compute Groups are built to accelerate intensive work using the hardware you already have access to. Any form of hardware can be connected to build self-sufficient compute grids. These resources can be on a single LAN, or geographically dispersed across a WAN.

The Distributed Computer

The Distributed Computer is your own private cloud, and makes the toughest jobs run in minutes.A global network where multiple compute grids are available through DCP. It allows you to access thousands of cores without managing any of the infrastructure.

How does DCP Work?

DCP is a complete distributed OS built to be standards-based, high performing, and future-proof.

Data Parallelism

The fundamental purpose of DCP is to execute the same task on different pieces of distributed data. To do this, the Compute API uses a memory safe generator called compute.for( ). A developer only needs to do two things to execute this code - identify their ideal unit of data parallelism, and write the corresponding methods to execute. Each distinct parallel unit of work in DCP is called a 'slice'. A slice contains the data related to the job, the methods that will be run on this data, and a miniscule amount of instructional overhead. Slices can be broadcast to a local machine, a software-defined compute cluster, or a globally distributed cloud federation. Every slice generally maps to a distinct thread on a CPU or GPU. Developers can prototype applications to run in parallel on their own computer for estimating work requirements before deploying it in production.

Hardware Abstraction

DCP uses a JavaScript engine for building seamless cross-platform clusters. Unlike Containers and VMs, this engine is lightweight and does not require the user to configure hardware and servers. DCP automatically manages infrastructure through this layer, making it serverless for the user. An instance of DCP is called a Worker. Each Worker is a Node.js daemon that execute computational work on a device's CPU and/or GPUs. Unlike VMsor Containers, starting a Worker is as easy as opening a web page. Compared to VMs and Containers, the Worker has:

> Lower memory requirements, since the JS engine is an ultra-lightweight program.
> Faster response times, including lower cold-start and application switching delays.
> Greater flexibility from being platform agnostic.
> Improved security from having no I/O capability.

Network Scheduling

DCP solves the problem of scheduling parallel workloads in a distributed environment with flexible, platform agnostic algorithms. These algorithms estimate, distribute, and confirm work on any cluster of machines. The DCP Scheduler is a server-side component that coordinates outstanding workloads with available compute resources. It records the resource status of connected nodes to:

> Broadcast time and space-sharing commands.
> Avoid performance issues such as thrashing. The scheduler takes user-defined factors into account, such as optimized speed, energy use, and even the level of CPU and memory capacity it should not exceed. Because the DCP worker issuitable for all environments, it can seamlessly schedule across everything from bare metal to Kubernetes deployments, and even different geographies.
